静态工厂的优点

使用静态工厂方法有如下好处：

  1. 工厂方法有名字，能够准确描述即将被创建的对象。如果使用多个带有不同参数的构造器，往往会导致用户在不阅读文档的情况下难以使用类。而静态工厂方法是有名字的，这使得它们能更好的描述用途。
  2. 静态工厂方法不必创建新实例。事实上，对于可以重复使用的对象或不可变对象，工厂模式可以使用预先构建好的实例，或者缓存起来以备后续使用。静态工厂可以严格控制类的实例化，这有助于：
    1. 实现单例模式；
    2. 确保某些类不可实例化；
    3. 确保某些值不可变的类不会存在两个具有相等的值的实例，即仅当 a==b 时，a.equals(b)才为true；
  3. 静态工厂方法可以返回原返回类型的子类（实现类），这使得在实例化对象时有了更多的灵活性。例如，可以通过接口返回特定的对象，却又不会使该对象的类成为公有类。这种技术在基于接口的框架中非常常用。
    在 Java 8 之前，接口不能有静态方法，因此接口的静态工厂方法通常放在一个名为 Types的不可实例化的伴生类中。例如，Java Collections集合接口。现在这一限制已成为历史，但是对于接口中静态方法依然必须是公有的，因此实现还是应该放在一个package-private的类中。
  4. 静态工厂所返回的对象的类可以随方法的参数不同而不同。事实上，只要是签名返回类型的子类型就可以，甚至可能会随发行版本的不同而不同。
  5. 静态工厂方法返回的对象，甚至可以是在编写代码时尚不存在的类。这种静态工厂方法的应用是Service Provider Framework的基础。

静态工厂的缺点

同时，静态工厂方法也有一些缺点：

  1. 如果类不含有public或protected的构造器，就不能实例化。当然这也可能是件好事：它鼓励程序员们多用组合，少用继承；
  2. 此外，程序员可能很难发现静态工厂方法的存在（在不查阅文档的情况下）。

常用的命名习惯

  • from——类型转换，通常只有一个参数
  • of——组合方法，带有多个参数
  • valueOf——更复杂的from和of的替代品
  • instance或getInstance——参数通常是设置项，常用于单例模式
  • create或newInstance——类似instance，但是会创建新实例
  • getType——用于可以提供多个类的工厂方法，Type指定工厂返回的类型
  • newType——类似getType，但会创建新实例
  • type——简化的getType和newType

总之，静态工厂方法和构造器各有好处，但是通常静态工厂更合适。

8. 避免使用 finalizer 方法和 cleaner 方法

我们都知道，finalizer方法并不是析构器。然而它比想象中的更糟糕：它通常具有不可预测的行为，极端危险，而且绝大多数情况下不需要使用。cleaner方法略好一些，但是还是很缓慢，而且也没有必要使用。在 Java 中，确保资源被释放的最佳实践是使用 try-with-resource 块或者 try-finally 块。

finalizer方法有以下问题：

  1. 不保证被及时执行。没有任何方法可以保证finalizer方法被及时执行。 而且，它的行为取决于JVM的实现，这使得同一程序在不同平台的行为可能大相径庭。
  2. 不保证被执行。事实上，Java 根本不保证finalizer会被执行！你绝对不能依赖finalizer或者cleaner方法来释放重要的资源。
  3. 如果忽略finalizer中被抛出的异常，会导致对象的回收过程被中止。未被捕获的异常使得对象处于被破坏的状态，这会导致任何不可预期的行为发生。此外，如果异常发生了finalizer方法中，甚至不会打印任何警告，也不会使线程终止。cleaner方法倒是没有这种问题。
  4. finalizer有严重的性能问题，它会阻止有效的垃圾回收，并大大拖慢程序的运行。cleaner好一点，不过也很慢。
  5. finalizer有严重的安全问题。构造器抛出的异常本可以阻止对象继续存在，但finalizer会妨碍这一点，并致使finalizer attack的存在。

如果你真的需要确保资源被释放，那么为你编写类实现AutoClosable接口，并要求使用者在使用完毕后调用close方法（比如使用try-with-resource块）就可以了。

那么finalizer和cleaner方法有什么好处呢？其实也是有的：

  1. 作为安全网。虽然不确保被执行，但是总比永远不被执行好。不过如果要这样用，请考虑是否值得付出性能代价，有没有更好的方法。
  2. 管理本地对等对象。本地对等对象并非普通 Java 对象，因此不会被垃圾回收。如果它没有关键资源又可以接受性能损耗的话，用cleaner方法再好不过了。

总之，大部分情况下不要使用它们。如果真的要用，至少也不要用finalizer方法。
