MapReduce 原理

2022-08-01#MapReduce

基本模型

MapReduce 的基本原理是利用一个 key-value 对来产生一个输出的 key-value 对集合。这个计算过程用两个函数表达这个计算：

  • Map 接受一个输入的 key-value 对，并生成一个中间 key-value 对集合。此后库函数会把所有具有相同中间 key 值的 value 聚合之后传递给 Reduce 函数。
  • Reduce 函数接受一个特定的中间 key 值及其相关的的 value 集合，并将它们聚合成一个较小的 value 集合（通常只产生 0 或 1 个输出）。
  • 通常使用迭代器传递中间 value 值，以处理可能无法全部放入内存的 value 集合。

实现

MapReduce 的实现取决于具体的环境。在 Google 内部广泛使用的一个实现是这样的：

  • 将 Map 处理的输入数据自动分割为 M 个数据片段的集合，并将 Map 调用分布到多台机器上执行，输入的数据片段也在不同的机器上并行处理。
  • 调用分区函数将 Map 产生的中间 key 值分成 R 个不同的分区，并将 Reduce 调用也分布到多台机器上执行。
    具体的全过程如下：
  1. 用户程序调用 MapReduce 库将输入文件分成 M 个数据片，其大小通常为 16~64 MB，可以由用户指定。
  2. 在集群中创建大量子程序，其中一个为 master 程序，其他为 work 程序。master 将任务分配给所有空闲的 worker。
  3. 被分配 Map 任务的 worker 读取相关的输入数据片段，并解析出key/value对，然后传给用户定义的 Map 函数，并将其生成的中间key/value对缓存在内存中。
  4. 缓存的key/value对通过分区函数分成 R 个区域，并周期性地写入本地磁盘。这些存储位置将被回传给 master，由 master 将这些存储位置传给 Reduce worker.
  5. Reduce worker 程序使用 RPC 从 Map worker 所在主机上的磁盘上读取缓存数据，并在读取完所有的中间数据后对 key 进行排序以聚合。如果中间数据太大无法在内存中排序，则外排。
  6. Reduce worker 遍历排序完后中间数据，并对每一个唯一的中间 key 值，将其对应的 value 集合传给用户定义的 Reduce 函数。其输出追加到所属区块文件的尾部。
  7. 所有任务完成后，master 唤醒用户程序，并返回 MapReduce 调用结果。
  8. 此时输出存在 R 个输出文件中，通常不需要合并：它们会被用作另一个 MapReduce 调用的输入，或者在其他可以处理分割后的文件的分布式应用中处理。
    Master 需要存储每一个 Map 和 Reduce 任务的状态，以及 Worker 机器的标识符。它就像一个数据管道，中间文件存储区域的位置信息通过这个管理从 Map 传递到 Reduce。对于每个完成的 Map 任务，master 存储了 Map 任务的所有中间文件存储区域的大小和位置。此后，master 会逐步地推送Reduce 任务给空闲的 worker.

容错机制

worker 故障

Master 会周期性地 ping 每个 worker。如果超时没有收到 worker 的返回信息，master 会把这个 worker 标记为失效，并把对应的 Map 任务重设为初始状态，然后安排给其他 worker。同样的，如果是 Reduce 任务也会被重置并等待重新调度。
当 worker 失效时，它对应的 Map 输出也会被标记为不可访问，因此即使它的 Map 任务已完成也需要重新执行。所有依赖它的 Reduce 任务的worker 都会被收到通知。它们将会从新的 worker 读取数据。
对于已完成的 Reduce 任务，由于其输出已经被存储在全局文件系统中了，所以不需要重新执行。

master 故障

Master 的重启较为麻烦，所以直接中止 MapReduce 运算，并由客户决定是否要重新执行 MapReduce 操作。

不一致处理

MapReduce 程序保证：

当用户的 Map 和 Reduce 操作都是输入确定性函数（即相同的输入产生相同的输出）时，任何情况下的输出都和程序没有出现错误、顺序执行产生的输出是一样的。

这个特性是通过 Map 和 Reduce 任务的输出的原子性来保证的。每个任务都会把自己的输出写入到私有的临时文件中，直到任务完成后才会发布到分布式系统里。
Map 任务完成后后把包含 R 个中间结果的临时文件的任务完成信息发给 master。master 接到信息后，如果该任务已完成，则会忽略该消息；否则会保存这 R 个文件的位置至自身存储中。
Reduce 任务完成后，worker 程序以原子的方式把临时文件重命名为最终的输出文件。如果 Reduce 任务在多台机器上执行，则会使用多次重命名操作来保证只有一个最终输出文件。

存储位置

在这种设计中，网络带宽是主要的资源瓶颈。因此，输入数据尽可能地被存储在集群中机器的本地磁盘上。GFS 把每个文件按 64 MB 一个 Block 分割并存储在多台机器上（通常是 3 份），而 MapReduce 在调度任务时会尽可能地把 Map 任务调度在包含相关输入数据拷贝的机器上执行。如果不行，也会尽可能地其附近的机器上执行。

任务粒度

如前所述，Map 被拆分成 M 个片段，把 Reduce 拆分成 R 个片段执行。理想情况下，M 和 R 应当比集群中 worker 的机器数量大得多，这样可以提高集群的动态负载均衡能力，而且也能加快故障恢复的速度。
但实际上，M 和 R 的取值受到内存使用的客观限制，而且 R 值还是由用户指定的，所以通常需要取一个合理的 M 值：16~64 MB 为宜，R 值则通常为欲使用的 worker 机器的倍数。常用的比例是 M=200000:R=5000:worker=2000(机器)。

备用任务

在 MapReduce 中存在严重的木板效应：影响总耗时的通常是最慢的那个机器。总有一台机器的速度远慢于其他机器，其原因是很复杂的：有时候是硬盘出了问题，有时候是初始化的 Bug，总之各式各样。
MapReduce 存在一个通用的机制来避免这种现象，当任务接近完成时，master 会调度备用进程执行剩下的 处理中 的任务，直到原本或备用的进程完成了任务。这样的机制只会多消耗百分之几的总计算资源，但能提高非常多的效率：在排序任务中，关闭备用任务机制后要多花44%的时间。

其余技术

调优分区函数

中间 key 需要使用分区函数来分区，然后再输入到后续任务中。默认是使用 hash 方法进行分区，不过有时候有更有用的方法。比如，输出的 key 值是 URL，而我们希望每个域名的条目都能放在同一个输出文件中，这时就需要提供专门的分区函数。

顺序保证

确保在每个分区中，中间 key/value 对的处理顺序是按 key 值增量顺序处理的，这样可以生成有序的输出文件，也便于排序。

聚合函数

有时候，Map 函数产生的中间 key 值数据会大量重复，而且用户定义的 Reduce 函数也满足结合律和交换律。这时可以允许用户指定一个聚合函数，它会在本地合并记录，然后再通过网络发送出去。
通常情况下，聚合函数和 Reduce 函数是一样的。
